

# Production Repair: Non-Contiguous Completions & Stale Timeline

## Root Cause (confirmed from data)

### Per-student analysis:

**Doğukan (Fatih, 2/week, rights=8)**
- Timeline: `[#3 Feb25 planned, #4 Feb26 planned, #1 Mar4 planned, #2 Mar5 planned, #5 Mar11 completed, #6 Mar12 planned, #7 Mar18 planned, #8 Mar19 planned]`
- Bug: 4 planned instances have dates BEFORE the completed instance (Mar11). These were generated by `ensureInstancesForWeek` for past weeks, inserting planned instances chronologically before the completed one.
- UI renders chronologically → box 5 is filled, boxes 1-4 empty. Shows "1/8" with gap.

**Emir (Fatih, 2/week, rights=8)**
- Timeline: `[#1 Feb24 ✓, #2 Feb26 ✓, #3 Mar3 planned, #4 Mar5 planned, #5 Mar10 ✓, #7 Mar17 ✓, #6 Mar12 planned, #8 Mar19 planned]`
- Bug: Planned instances #3 (Mar3) and #4 (Mar5) sit between completed instances, creating gaps at display positions 3,4,6.
- Also: Emir's template changed (from different times), but old planned instances kept old dates mixed with new completions.

**Yiğit (Eren, 1/week → now 2/week)**
- `lessons_per_week=1` in tracking but `student_lessons` has 2 templates. So `total_rights = 1×4 = 4` but `templateCount = 2 → total=8`.
- Has 5 completed, 0 planned. `getRemainingRights` uses templateCount(2)→total=8, so shows "5/8". But LessonTracker uses `templateCount` for grid size (8 boxes) while only 5 instances exist → last 3 boxes have no instance data.

### Systemic root cause

1. **`ensureInstancesForWeek` creates planned instances for ANY week ≥ today, even if that date falls BEFORE the student's last completed instance**. This inserts planned lessons in the middle of the completed timeline.

2. **`lessons_per_week` in `student_lesson_tracking` becomes stale** when templates are changed. `EditStudentDialog` updates it only on the first save path but `getRemainingRights`, `ensureInstancesForWeek`, and RPCs use different sources for "weekly count" (some use tracking, some use template count).

3. **`lesson_number` was assigned at creation time** and never re-sequenced when dates changed, making `lesson_number` meaningless for display ordering.

4. **Pre-sequential-RPC completions** happened out of chronological order, creating permanent gaps that no amount of renumbering fixes.

## Fix Strategy

### Code Fix 1: `ensureInstancesForWeek` — never generate before last completed
**File: `src/hooks/useScheduleGrid.ts`**

After computing `lessonDate` for a new planned instance, skip if that date is ≤ the student's last completed instance date in the current cycle. This prevents creating planned instances in the timeline gap.

Add per-student query for max completed lesson_date. Skip generation for dates at or before that date.

### Code Fix 2: Sync `lessons_per_week` on every template save
**File: `src/components/EditStudentDialog.tsx`**

The existing code already updates `lessons_per_week` in some paths. Ensure it always updates `student_lesson_tracking.lessons_per_week` with the new `lessonsPerWeek` value on every save, not just when creating tracking records.

### Data Repair Migration (single SQL migration)

The migration does the following in order:

**Step 1: Sync `lessons_per_week`**
```
UPDATE student_lesson_tracking slt
SET lessons_per_week = sub.cnt
FROM (
  SELECT student_id, teacher_id, count(*) as cnt
  FROM student_lessons
  GROUP BY student_id, teacher_id
) sub
WHERE slt.student_id = sub.student_id AND slt.teacher_id = sub.teacher_id
  AND slt.lessons_per_week <> sub.cnt;
```

**Step 2: Delete planned instances that sit before the last completed instance**
For each active student, find `max_completed_date`. Delete all planned instances with `lesson_date <= max_completed_date` in the current cycle. This removes the "gap-causing" early planned instances.

**Step 3: Regenerate missing planned instances**
For students where `completed + remaining_planned < total_rights`, generate new planned instances starting from the day after the last existing instance (completed or planned), using the student's template slots.

**Step 4: Renumber `lesson_number` to chronological order**
For all current-cycle instances per student, set `lesson_number = ROW_NUMBER() OVER (ORDER BY lesson_date, start_time)`. This makes lesson_number match display position.

**Step 5: Log and report**
- Log all changes to `balance_events` with `event_type = 'data_repair'`
- RAISE NOTICE for each student modified
- Report students where completed count > total_rights (manual review)

### Safety constraints
- Never delete or modify completed instances
- Never touch teacher_balance
- Never touch archived students
- All planned deletions are only for instances with dates BEFORE the last completed date
- Regenerated planned instances get correct `package_cycle`
- Over-completed students (like Yiğit with 5/4 before sync) become normal after `lessons_per_week` sync (5/8 after sync)
